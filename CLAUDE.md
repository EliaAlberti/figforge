## Repository Structure

```
figforge/
├── .claude-plugin/
│   ├── plugin.json        ← Plugin manifest (v2.0.0)
│   └── marketplace.json   ← Marketplace metadata for private distribution
├── agents/                ← 6 specialist agents with YAML frontmatter
├── skills/                ← 10 self-contained skills (incl. init)
├── templates/             ← Sprint config + verify.sh (legacy template ref)
├── persistence/           ← Beads + sprint + task sizing docs
├── scripts/               ← install.sh (manual install path)
├── docs/                  ← Documentation files
└── data/                  ← Research reports
```

## Distribution Paths

FigForge supports two installation paths:

1. **Plugin install (recommended)** — users run `/plugin marketplace add EliaAlberti/figforge` then `/plugin install figforge`. The `.claude-plugin/plugin.json` manifest drives this.
2. **Manual install** — users clone the repo and run `bash scripts/install.sh`. Copies agents and skills to `~/.claude/`.

Both paths result in the same capabilities available in Claude Code.

## Development Rules

- **No iOS references** — this is a Figma-only framework
- **No marketing content** — that belongs in SwiftForge
- **Every agent must handle Remote Mode gracefully** — Local Mode is not guaranteed
- **Agent bodies are verbatim** — users may spawn them as agent team members
- **Skills are self-contained** — each SKILL.md must work independently
- **Never overwrite a user's CLAUDE.md** — the init skill appends, never replaces

## Testing Changes

After editing agents or skills, verify locally:

```bash
# Manual install path
bash scripts/install.sh

# Plugin path (test in Claude Code)
/plugin marketplace add /path/to/figforge
/plugin install figforge
```

Then test in a project with `/figforge:init` followed by `"spin up the Figma team"`.

## Versioning

- `plugin.json` version follows semver
- Bump minor for new skills/agents, major for breaking structure changes
- Tag releases: `git tag v2.0.0 && git push --tags`
